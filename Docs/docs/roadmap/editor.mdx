# Editor Plan

SafiEngine already ships a MicroUI debug overlay with a Scene hierarchy, an editable Inspector (numbers, vec3s, checkboxes, dropdowns), physics components, and an engine-owned render stage. The editor is the natural extension: the same in-process UI grows into a tool that lets you **build, play, and ship a game without writing C bootstrap code for every scene**.

The editor is **not a separate application**. It is a set of systems and panels that run in the same process as the game, behind the same `enable_debug_ui` flag that already gates the Inspector.

## Design principles

- **One binary, two modes** ‚Äî "Edit" and "Play". Edit mode pauses fixed update and game systems; Play mode runs them. No separate editor process, no IPC.
- **Data in ECS, not in the editor** ‚Äî every piece of scene state lives in flecs components. The editor only reads/writes components; it never keeps a parallel model of the world.
- **Undo = snapshot** ‚Äî undo/redo restores a serialized snapshot of affected entities rather than reverse-applying ops. Simpler, fewer edge cases, and it composes with physics state.
- **Stay in C** ‚Äî the editor is implemented in the same engine C code, using MicroUI and the existing render stack. No ImGui, no C++ bindings, no second UI toolkit.
- **Author scenes as files** ‚Äî the source of truth for a level is a scene file on disk, not code. Games ship as `engine_binary + assets/ + scenes/`.

## Current foundation (already shipped)

- **Scene hierarchy panel** ‚Äî entities listed by `SafiName`, nested by `EcsChildOf`, click-to-select. ([Debug UI](/api/ui/debug_ui))
- **Inspector panel** ‚Äî collapsible sections per component, editable number / vec3 / checkbox / dropdown widgets. Covers Transform, Camera, MeshRenderer, Spin, RigidBody, Collider, and all 5 light types. Drops stale selection (`ecs_is_alive`) across scene reloads.
- **Engine-owned render stage** ‚Äî the render pipeline is a set of ECS systems on the render phase; editor systems can insert overlays (gizmos, wireframes) without forking the renderer. ([Render Overview](/api/render/overview))
- **Fixed timestep** ‚Äî `SafiFixedUpdate` is separate from the variable update, so pausing physics is just "don't run fixed update". ([Scheduler](/api/core/scheduler))
- **Physics integration** ‚Äî Jolt bodies driven by `SafiRigidBody` + `SafiCollider`, transform sync in both directions. ([Physics](/api/ecs/physics))
- **Scene serialization** ‚Äî `safi_scene_save` / `safi_scene_load` / `safi_scene_clear` plus in-memory `safi_scene_snapshot_all` / `safi_scene_restore_snapshot` for non-destructive Play‚ÜíStop round-trips. ([Scene](/api/scene/scene))
- **Editor-mode singleton** ‚Äî `SafiEditorState { mode, selected_entity }` with `Edit / Play / Paused`. The `safi_app_tick` loop gates the fixed + game pipelines on `mode == PLAY`; variable + render always run. ([Editor State](/api/editor/editor_state))
- **Dedicated game phase** ‚Äî user gameplay opts into `SafiGamePhase` (distinct from `EcsOnUpdate`) so it freezes in Edit mode while engine-owned variable systems keep ticking. ([Scheduler](/api/core/scheduler))
- **Editor fly-cam** ‚Äî `SafiEditorCamera` entity with WASD/QE translate + right-mouse rotate. Installed automatically with the debug UI; arbitrates the `SafiActiveCamera` tag with the gameplay camera on every Edit ‚Üî Play switch. ([Editor Camera](/api/editor/editor_camera))
- **Gizmo draw list** ‚Äî `safi_gizmo_draw_line / _box_wire / _aabb / _sphere_wire`, drained once per frame inside the main render pass. The substrate M4 gizmo handles sit on. ([Gizmo Draw List](/api/render/gizmo))
- **Camera pose + screen-ray** ‚Äî `SafiCamera` carries explicit eye/forward/up; `safi_camera_screen_ray` unprojects a mouse click into a world-space ray so picking and gizmo drag share one convention. ([Camera Math](/api/render/camera))
- **Editor keybinds** ‚Äî F1 toggles Edit ‚Üî Play, F6 snapshots the world, F7 restores the snapshot. Engine-owned stand-ins for the M1 toolbar; skipped while a MicroUI field has focus. ([Editor Shortcuts](/api/editor/editor_shortcuts))

## Milestones

### M1 ‚Äî Play / Pause / Stop (foundation) ‚Äî **engine side shipped, UI pending**

The single most important editor feature. Without it, nothing else feels like an editor.

- **Toolbar panel** with three buttons: `‚ñ∂` Play, `‚è∏` Pause, `‚èπ` Stop.
- **Edit mode (default)**: fixed update is skipped; gameplay systems (user systems registered via a new `SafiGamePhase` tag) are skipped; render + transform propagation still run so the scene is visible; physics bodies are paused but their components stay editable in the Inspector.
- **Play mode**: before entering, snapshot every entity that carries at least one user-mutable component (Transform, RigidBody velocity, etc.) into a side buffer. Enable fixed update and game systems.
- **Stop**: disable fixed update, restore entities from the snapshot. Pause just toggles the schedule without snapshotting.

### M2 ‚Äî Scene serialization

Save / load what's in the world to disk so work survives a restart.

- **Format**: JSON, one file per scene. Each entry = `{ name, parent?, components: { SafiTransform: {...}, SafiMeshRenderer: { model: "asset://meshes/box.glb" }, ... } }`.
- **Mesh / material references** are string asset URIs, resolved through the asset handle system (M3).
- **Writer**: walks every entity with a `SafiName`, emits components via a per-component registry `{ id ‚Üí to_json / from_json }`.
- **Reader**: creates entities, looks up asset URIs, attaches components in dependency order (Transform before GlobalTransform, RigidBody before Collider registration, etc.).
- **Not in scope**: cross-scene references, binary format, versioning. A schema bump = "bump the version field and rewrite the loader".

**Menu**: `File ‚Üí New Scene / Open‚Ä¶ / Save / Save As‚Ä¶` in a top menu bar.

### M3 ‚Äî Asset handle system & asset browser

Scenes must reference assets by a stable id, not by a live pointer.

- **`SafiAssetHandle<T>`** ‚Äî opaque 64-bit id, one slot per asset type (mesh, material, texture, shader). Handles are resolvable through a central registry that caches loaded assets and deduplicates by path.
- **Asset browser panel** ‚Äî grid or tree view of the `assets/` directory. Double-click to preview (mesh in a mini viewport; texture as a swatch). Drag a row onto an Inspector field to assign.
- **Hot-reload** hook: watch `assets/` with SDL's filesystem notifications; on change, re-upload the GPU resource and patch every handle. Out of scope for MVP; stub the API.

### M4 ‚Äî Viewport gizmos & editor camera

Direct manipulation in the 3D viewport ‚Äî the part that makes the editor feel like an editor rather than a form filler.

- **Editor camera** ‚Äî a fly-cam with WASD + right-mouse-look, owned by a dedicated `SafiEditorCamera` entity. Becomes the active camera when in Edit mode; Play mode switches to whatever entity has `SafiActiveCamera`.
- **Translate gizmo** ‚Äî three axis-aligned arrows at the selected entity's world position. Drag an arrow ‚Üí raycast from mouse onto the axis line, update `Transform.position`.
- **Rotate gizmo** ‚Äî three orthogonal rings; drag tangentially to spin the quaternion.
- **Scale gizmo** ‚Äî three axis handles ending in cubes; drag along axis.
- **Mode switch**: `Q / W / E / R` for select / translate / rotate / scale, matching Unity/Unreal muscle memory.
- **Rendering** ‚Äî gizmos are drawn by a dedicated editor render system that runs after opaque geometry and reads/writes depth selectively (so handles aren't occluded).

### M5 ‚Äî Entity creation & component add/remove

Authoring entities from scratch without C code.

- **Create menu** (`+` button in the Scene panel or right-click context menu):
  - `Empty` ‚Äî just `SafiTransform + SafiName`.
  - `Mesh` ‚Äî empty + `SafiMeshRenderer` (prompts for a mesh asset).
  - `Light ‚Üí Directional / Point / Spot / Rect / Sky`.
  - `Camera` ‚Äî `SafiTransform + SafiCamera`.
  - `Physics Body ‚Üí Static Box / Dynamic Sphere / ‚Ä¶` ‚Äî transform + rigidbody + collider presets.
- **Component panel** ‚Äî each Inspector component gets an `√ó` in its header to remove it; bottom of Inspector has `+ Add Component‚Ä¶` that opens a searchable list of every registered component.
- **Parent assignment** ‚Äî drag an entity onto another in the Scene panel to re-parent; drag to the root area to unparent.

### M6 ‚Äî Undo / redo

Snapshot-based, scoped to a small window of recent changes.

- Every Inspector edit, gizmo drag, entity create/delete, and parent change pushes a snapshot of the affected entities onto a bounded ring buffer (say 64 steps).
- `Cmd/Ctrl-Z` / `Cmd/Ctrl-Shift-Z` pop/push between stacks.
- Gizmo drags coalesce ‚Äî one snapshot at drag-start, one at drag-end, nothing in between.

### M7 ‚Äî Prefabs / multi-select / polish

- **Multi-select** in the Scene panel with Cmd/Shift; Inspector shows shared fields only, edits apply to all.
- **Prefabs** ‚Äî save a subtree as `*.prefab.json`; instantiate by drag-and-drop into a scene. Changes to the prefab propagate unless overridden.
- **Debug physics wireframe** ‚Äî hook Jolt's debug renderer into the editor render system.
- **Performance panel** ‚Äî frame time, draw call count, physics step time, entity count.

### M8 ‚Äî Build & ship

The part that closes the loop from "I edited something" to "I can hand this to a player".

- **`File ‚Üí Build‚Ä¶`** packages: the engine binary, the `assets/` folder (only files referenced by any scene), the `scenes/` folder, and a tiny `bootstrap.json` naming the first scene.
- The produced binary launches with `enable_debug_ui = false`, loads `bootstrap.json`, and plays immediately.
- Platforms in scope for MVP: macOS (Intel + ARM). Windows and Linux follow the same pattern through SDL3.

## Non-goals (for now)

- **Visual scripting / node graphs.** Gameplay stays in C. The editor is a level/entity tool, not a logic tool.
- **Terrain, animation timeline, particle editors.** These are subsystems that need to exist in the engine first; the editor gets them when the engine does.
- **Multi-user / remote editing.** Single user, local files.
- **A separate "Game" binary.** Games ship as the same engine binary in non-debug mode plus data.

## Open questions

- **Gizmo interaction math** ‚Äî whether to use screen-space handle picking (simple, slightly imprecise) vs. ray/line intersection (more code, exact).
- **Undo granularity for physics** ‚Äî if the user scrubs `RigidBody.mass` during Play, should that be snapshotted? Probably yes in Edit, never in Play.
- **Asset URI scheme** ‚Äî `asset://meshes/box.glb` vs. plain relative paths. Asset URIs are more flexible (multiple roots) but add indirection.
- **Which component registry?** A macro-based one that every component opts into, or one centralized file that hand-lists components. Macro scales better but complicates the public header.

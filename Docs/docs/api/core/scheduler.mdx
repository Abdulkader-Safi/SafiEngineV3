# Scheduler & Stages

```c
#include <safi/ecs/ecs.h>
```

SafiEngine runs one flecs pipeline per frame. Systems are assigned to **phases** that execute in a fixed order. This is the engine's scheduling model — lightweight, implicit, and built on flecs pipelines rather than a custom scheduler.

## Current phases

```text
EcsOnLoad     → input pumped, SafiTime updated
EcsOnUpdate   → gameplay / logic systems (yours)
EcsOnStore    → rendering (currently demo-owned)
```

You can also use `EcsPreUpdate`, `EcsPostUpdate`, and `EcsPreStore` — see the [flecs quickstart](https://www.flecs.dev/flecs/md_docs_2Quickstart.html).

## Registering a system

```c
ECS_SYSTEM(world, my_system, EcsOnUpdate, SafiTransform);
```

Or with explicit dependency:

```c
ecs_system(world, {
    .entity = ecs_entity(world, { .name = "MySystem", .add = ecs_dependson(EcsOnUpdate) }),
    .query.terms = { { ecs_id(SafiTransform) } },
    .callback = my_system,
});
```

:::warning WIP — Scheduling gaps
The flecs pipeline phases are available, but the engine does not yet provide:
- **Startup stage** — no one-shot startup system registration API; initialization runs in `main()` before `safi_app_run`
- **Fixed update stage** — no fixed-timestep accumulator or `FixedUpdate` schedule; all systems run at variable `dt`
- **Engine-owned render stage** — the render system currently lives in the demo, not in the engine
- **Higher-level ordering API** — only raw flecs `ecs_dependson` phase ordering is available; no explicit before/after system dependencies
:::
